# Rasa Pipeline & Policies — Πλήρης Οδηγός

## Επισκόπηση

Η αρχιτεκτονική του Rasa χωρίζεται σε δύο βασικά μέρη:

- **Pipeline (NLU)** — Επεξεργάζεται κάθε μεμονωμένο μήνυμα του χρήστη: *"Τι είπε ο χρήστης;"*
  Παράγει **intent**, **entities** και **confidence scores**.
- **Policies (Core/Dialogue)** — Λαμβάνουν την έξοδο του NLU *μαζί με το ιστορικό της συνομιλίας* και αποφασίζουν:
  *"Τι πρέπει να κάνει το bot στη συνέχεια;"* Προβλέπουν την **επόμενη ενέργεια (action)**.

```
Μήνυμα χρήστη → Pipeline (NLU) → intent + entities → Policies (Core) → επόμενη ενέργεια
```

## Μέρος 1 — NLU Pipeline

### Οπτικοποίηση: Πώς μεταμορφώνεται ένα μήνυμα βήμα-βήμα

Ας πάρουμε το μήνυμα: **`"Θέλω να κλείσω πτήση για Θεσσαλονίκη"`**

```
┌─────────────────────────────────────────────────────────────────────────┐
│  ΕΙΣΟΔΟΣ (raw text)                                                     │
│  "Θέλω να κλείσω πτήση για Θεσσαλονίκη"                                │
└───────────────────────────────┬─────────────────────────────────────────┘
                                │
                                ▼
┌─────────────────────────────────────────────────────────────────────────┐
│  1. WhitespaceTokenizer                                                 │
│  Χωρίζει το κείμενο σε tokens (λέξεις) με βάση τα κενά.                │
│                                                                         │
│  ➜ ["Θέλω", "να", "κλείσω", "πτήση", "για", "Θεσσαλονίκη"]            │
└───────────────────────────────┬─────────────────────────────────────────┘
                                │
                                ▼
┌─────────────────────────────────────────────────────────────────────────┐
│  2. RegexFeaturizer                                                     │
│  Ελέγχει αν κάποιο token ταιριάζει με regex patterns ή lookup tables    │
│  που έχουν οριστεί στα training data (π.χ. λίστα πόλεων).              │
│                                                                         │
│  ➜ ["Θέλω", "να", "κλείσω", "πτήση", "για", "Θεσσαλονίκη"✓regex]     │
│                                                        ↑                │
│                                              ταιριάζει με pattern       │
│                                              πόλεων → σημαία ON         │
└───────────────────────────────┬─────────────────────────────────────────┘
                                │
                                ▼
┌─────────────────────────────────────────────────────────────────────────┐
│  3. LexicalSyntacticFeaturizer                                          │
│  Προσθέτει χαρακτηριστικά βάσει λεξικών/συντακτικών ιδιοτήτων:          │
│  κεφαλαίο γράμμα; αριθμός; τι υπάρχει πριν/μετά;                       │
│                                                                         │
│  ➜ "Θεσσαλονίκη" → [αρχίζει_με_κεφαλαίο=1, είναι_αριθμός=0,          │
│                       προηγούμενο_token="για", ...]                     │
└───────────────────────────────┬─────────────────────────────────────────┘
                                │
                                ▼
┌─────────────────────────────────────────────────────────────────────────┐
│  4. CountVectorsFeaturizer (επίπεδο λέξεων)                             │
│  Δημιουργεί bag-of-words αναπαράσταση — μετράει πόσες φορές             │
│  εμφανίζεται κάθε λέξη.                                                 │
│                                                                         │
│  ➜ {Θέλω:1, να:1, κλείσω:1, πτήση:1, για:1, Θεσσαλονίκη:1}           │
│     → αραιό διάνυσμα (sparse vector) [0,0,1,0,1,0,0,1,...]             │
└───────────────────────────────┬─────────────────────────────────────────┘
                                │
                                ▼
┌─────────────────────────────────────────────────────────────────────────┐
│  5. CountVectorsFeaturizer (character n-grams, char_wb, 1-4)            │
│  Σπάει κάθε λέξη σε υπο-ακολουθίες χαρακτήρων (1 έως 4).               │
│  Αυτό βοηθάει στην αντιμετώπιση τυπογραφικών λαθών.                     │
│                                                                         │
│  ➜ "κλείσω" → ["κ", "κλ", "κλε", "κλεί", "λ", "λε", "λεί",          │
│                  "λείσ", "ε", "εί", "είσ", "είσω", ...]               │
│                                                                         │
│  Ακόμα κι αν ο χρήστης γράψει "κλίσω" (τυπογραφικό), τα               │
│  περισσότερα n-grams θα ταιριάζουν → ανθεκτικότητα σε λάθη!            │
└───────────────────────────────┬─────────────────────────────────────────┘
                                │
                                ▼
┌─────────────────────────────────────────────────────────────────────────┐
│  6. DIETClassifier (Dual Intent and Entity Transformer)                 │
│  Το κεντρικό μοντέλο. Παίρνει ΟΛΑ τα features από τα προηγούμενα       │
│  βήματα και κάνει ταυτόχρονα:                                           │
│    • Ταξινόμηση intent                                                  │
│    • Εξαγωγή entities                                                   │
│  100 epochs, constrain_similarities=true                                │
│                                                                         │
│  ➜ intent: "κράτηση_πτήσης" (confidence: 0.92)                         │
│  ➜ entities: [{value:"Θεσσαλονίκη", entity:"πόλη", start:28, end:40}]  │
└───────────────────────────────┬─────────────────────────────────────────┘
                                │
                                ▼
┌─────────────────────────────────────────────────────────────────────────┐
│  7. EntitySynonymMapper                                                 │
│  Αντιστοιχίζει εξαγόμενες τιμές entities σε κανονικοποιημένες μορφές    │
│  βάσει συνωνύμων στα training data.                                     │
│                                                                         │
│  ➜ "Θεσ/νίκη" → "Θεσσαλονίκη"                                         │
│  ➜ "ΣΚΓ"      → "Θεσσαλονίκη"                                         │
└───────────────────────────────┬─────────────────────────────────────────┘
                                │
                                ▼
┌─────────────────────────────────────────────────────────────────────────┐
│  8. ResponseSelector                                                    │
│  Χειρίζεται retrieval intents (π.χ. FAQ, chitchat). Επιλέγει την        │
│  καλύτερη απάντηση από ένα σύνολο υποψηφίων.                            │
│  100 epochs, constrain_similarities=true                                │
│                                                                         │
│  ➜ Αν intent = "faq/ωράριο" → επιλέγει: "Είμαστε ανοιχτοί 9πμ-5μμ"   │
└───────────────────────────────┬─────────────────────────────────────────┘
                                │
                                ▼
┌─────────────────────────────────────────────────────────────────────────┐
│  9. FallbackClassifier                                                  │
│  Δίχτυ ασφαλείας. Ελέγχει:                                              │
│    • confidence < 0.3 → fallback                                        │
│    • διαφορά 1ου-2ου intent < 0.1 (αμφιβολία) → fallback               │
│                                                                         │
│  ➜ Αν confidence=0.92 και κανένα πρόβλημα αμφιβολίας:                   │
│     ΤΕΛΙΚΟ ΑΠΟΤΕΛΕΣΜΑ → intent: "κράτηση_πτήσης"                       │
│                          entity: "πόλη" = "Θεσσαλονίκη"                 │
│                                                                         │
│  ➜ Αν confidence=0.25:                                                  │
│     ΤΕΛΙΚΟ ΑΠΟΤΕΛΕΣΜΑ → intent: "nlu_fallback" (ζητάει διευκρίνιση)    │
└─────────────────────────────────────────────────────────────────────────┘
```

### Συνοπτικά

```
Ακατέργαστο κείμενο
   → tokenize (χωρισμός σε λέξεις)
      → featurize (αριθμητική αναπαράσταση)
         → classify + extract (ταξινόμηση intent, εξαγωγή entities)
            → normalize (κανονικοποίηση entities)
               → safety net (fallback αν υπάρχει αβεβαιότητα)
```

## Μέρος 2 — Policies (Πολιτικές Διαλόγου)

Οι πολιτικές λαμβάνουν το αποτέλεσμα του NLU pipeline (intent + entities) **μαζί με
το ιστορικό της συνομιλίας** και προβλέπουν ποια θα είναι η **επόμενη ενέργεια** του bot.

### 1. MemoizationPolicy

Απομνημονεύει **ακριβείς ακολουθίες** συνομιλιών από τα training stories.
Αν η τρέχουσα συνομιλία ταιριάζει *ακριβώς* με κάποιο story που έχει δει,
προβλέπει την επόμενη ενέργεια με πλήρη βεβαιότητα.

Σκεφτείτε το σαν **πίνακα αναζήτησης** — γρήγορο και αξιόπιστο για γνωστά
μονοπάτια, αλλά δεν μπορεί να γενικεύσει σε καινούριες καταστάσεις.

### 2. RulePolicy

Χειρίζεται **αυστηρούς, ντετερμινιστικούς κανόνες** που ορίζονται στο `rules.yml`.
Πρόκειται για σύντομα, μη-διαπραγματεύσιμα patterns, όπως:

- Πάντα απάντα στο "αντίο" με `utter_goodbye`
- Πάντα εκτέλεσε fallback action όταν ενεργοποιηθεί ο `FallbackClassifier`
- Πάντα ενεργοποίησε ένα form όταν ανιχνευθεί συγκεκριμένο intent

Οι κανόνες έχουν **προτεραιότητα πάνω από τα stories** και δεν εξαρτώνται
από το ιστορικό της συνομιλίας.

### 3. UnexpecTEDIntentPolicy

Μοντέλο βασισμένο σε Transformer που σημαίνει (flag) πότε ο χρήστης λέει
κάτι **απρόσμενο** δεδομένου του πλαισίου της συνομιλίας.

Δεν προβλέπει απευθείας την επόμενη ενέργεια — αντ' αυτού, ενεργοποιεί
το `action_unlikely_intent` όταν το intent του χρήστη δεν ταιριάζει με αυτό
που θα περίμενε σε εκείνο το σημείο του διαλόγου.

- **max_history: 5** — κοιτάζει τα τελευταία 5 βήματα
- **epochs: 100** — εκπαιδεύεται για 100 εποχές

### 4. TEDPolicy (Transformer Embedding Dialogue)

Η κύρια πολιτική μηχανικής μάθησης. Μαθαίνει να **γενικεύει** από τα
training stories, ώστε να χειρίζεται συνομιλίες που δεν έχει δει αυτολεξεί
(σε αντίθεση με το `MemoizationPolicy`).

- **max_history: 5** — λαμβάνει υπόψη τα τελευταία 5 βήματα
- **epochs: 100** — εκπαιδεύεται για 100 εποχές
- **constrain_similarities: true** — βελτιώνει τη βαθμονόμηση confidence

## Πώς συνεργάζονται οι Policies

Όταν πολλαπλές πολιτικές προβλέπουν μια ενέργεια, το Rasa χρησιμοποιεί
ένα **σύστημα προτεραιοτήτων** για να επιλύσει συγκρούσεις:

```
  Προτεραιότητα        Πολιτική                     Ρόλος
  ─────────────────────────────────────────────────────────────
  ▲ Υψηλότερη          RulePolicy                   Σκληροί κανόνες — κερδίζουν πάντα
  │                     MemoizationPolicy            Ακριβή ταιριάσματα stories
  │                     UnexpecTEDIntentPolicy        Σημαίνει απρόσμενα intents
  ▼ Χαμηλότερη          TEDPolicy                    Γενικευμένη ML πρόβλεψη
```

Αν ο `RulePolicy` έχει πρόβλεψη, κερδίζει. Διαφορετικά, χρησιμοποιείται
η πρόβλεψη με το υψηλότερο confidence ανάμεσα στις υπόλοιπες πολιτικές.

Αυτή η πολυεπίπεδη προσέγγιση προσφέρει:
- **Ντετερμινιστικό έλεγχο** όπου χρειάζεται (rules)
- **Αξιόπιστη ανάκληση** εκπαιδευμένων μονοπατιών (memoization)
- **Ευέλικτη γενίκευση** σε νέες καταστάσεις (TED)
